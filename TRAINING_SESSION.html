Project Overview

What the system does, the problem it solves, and the core user experience

01 — Project Overview

What problem does this solve?

Knowledge workers spend a significant part of their day writing repetitive replies to emails and Slack messages. This system monitors your Gmail inbox and Slack workspace in real-time and generates an AI-powered draft reply for every message you receive — ready to review, tweak, and send in seconds.

📧
Gmail Drafts
Polls your inbox every minute. Filters promotional / automated mail. Creates a Gmail draft in the same thread using GPT-4o.
💬
Slack DM Drafts
Polls every 30 seconds for new direct messages sent to you. Delivers an AI draft privately via the bot DM.
📢
Slack Channel Drafts
Polls every 30 seconds for new messages in channels you belong to. No need to invite the bot — uses your personal token.

Core User Experience

Step 1Someone sends you a message
→
Step 2System detects it (poller / event)
→
Step 3GPT-4o generates a draft reply
→
Step 4Draft delivered to you privately
→
Step 5You copy, adjust & send

Key Design Principles

  • Non-intrusive: The bot never sends anything on your behalf. It only creates drafts for you to review.
  • Zero false positives: Smart filtering means only real human-to-human messages get drafted.
  • Deduplication: Every message is processed exactly once — no duplicate drafts ever.
  • Context-aware: The AI reads conversation history and thread replies before generating a draft.
  • Multi-user ready: Multiple users can connect their own Gmail / Slack accounts via OAuth.
  • No public URL needed: Slack uses Socket Mode (WebSocket) — no ngrok, no tunnels.

Two Separate Token Types (Critical Concept)

🤖 Bot Token (xoxb-…)
  • Used to send messages as the AI Assistant bot
  • Delivers draft notifications via bot DM to you
  • Stored in .env as SLACK_BOT_TOKEN
  • One token for the whole workspace
👤 User Token (xoxp-…)
  • Used to read messages on behalf of a user
  • Read DMs, channel history, user profile
  • Stored in DB per workspace (via OAuth /slack/connect)
  • One token per connected Slack user

Gmail Integration

OAuth flow, email filtering, deduplication, and AI draft creation

05 — Gmail Integration (Part 1): OAuth & Polling

Gmail OAuth Flow

  1. User visits GET /auth/google in browser — redirected to Google consent screen.
  2. After consent, Google redirects to GET /auth/google/callback with an auth code.
  3. Server exchanges the code for access_token + refresh_token.
  4. Tokens saved to the Token table in the DB, linked to the user's Google email (userId).
  5. When the access token expires, tokenService.ts automatically refreshes it using the refresh token.

Email Polling Pipeline (emailPoller.ts)

Cron: 1 minTrigger poll
→
Gmail APIFetch 30 unread emails
→
Dedup #1DB: messageId unique check
→
Dedup #2DB: threadId check per user
→
FilteremailUtils.ts rules
→
OpenAIGenerate draft reply
→
Gmail APICreate draft in thread

Why Two Layers of Deduplication?

Dedup #1 — Message-level (processedEmail table)

A messageId unique constraint in the DB. Even if the poller runs twice, the second attempt fails fast with "already processed" — no duplicate drafts.

Dedup #2 — Thread-level (draftLog table)

Before drafting, we check if a draft already exists for this threadId for this user. Prevents drafts for every reply in a long email chain.

05 — Gmail Integration (Part 2): Email Filtering

Why filtering matters

Without filtering, the AI would waste API credits drafting replies to Amazon shipping notifications, GitHub CI build failures, and promotional newsletters. The filter is layered — any one rule can reject an email:

#Filter LayerWhat it catchesFile
1 Gmail Label Rejects CATEGORY_PROMOTIONS & SPAM — Gmail's own ML classifier. Very reliable for ads & spam. emailUtils.ts
2 List-Unsubscribe Header Any email with a List-Unsubscribe header is bulk mail. Real human emails never have this. emailPoller.ts
3 Sender Pattern Regex patterns: no-reply@, noreply, mailer-daemon, newsletter@, notifications@, etc. emailUtils.ts
4 Subject Pattern [automated], % off, flash sale, build failed/passed, dependabot, shopping recommendations, etc. emailUtils.ts
✅ What is intentionally NOT filtered
  • CATEGORY_UPDATES — Real transactional emails (bank statements, invoice from a person) land here
  • CATEGORY_SOCIAL — LinkedIn messages, real social notifications with a human behind them
  • Company domain emails (e.g., john@company.com) — real people

Sample: Email Filter Code Flow

src/utils/emailUtils.ts — shouldSkipEmail()
export function shouldSkipEmail(
  sender: string, subject: string,
  labels: string[] = [], hasListUnsubscribe: boolean = false
): EmailFilterResult {

  // 1. Gmail category labels (CATEGORY_PROMOTIONS, SPAM)
  for (const label of labels) {
    if (SKIP_LABEL_PATTERNS.some(p => p.test(label)))
      return { skip: true, reason: `gmail_label:${label}` };
  }

  // 2. Bulk mail indicator header
  if (hasListUnsubscribe) return { skip: true, reason: 'list_unsubscribe_header' };

  // 3. Sender pattern (no-reply, newsletter@, etc.)
  for (const pattern of AUTOMATED_SENDER_PATTERNS) {
    if (pattern.test(sender))
      return { skip: true, reason: `sender_pattern` };
  }

  // 4. Subject pattern (flash sale, % off, etc.)
  for (const pattern of AUTOMATED_SUBJECT_PATTERNS) {
    if (pattern.test(subject))
      return { skip: true, reason: `subject_pattern` };
  }

  return { skip: false }; // ← Real email: proceed to AI drafting
}

06 — Slack Integration (Part 1): Connection & Pollers

Why Socket Mode instead of HTTP webhooks?

Socket Mode = WebSocket connection from your server TO Slack
  • No public URL required — works from localhost, behind firewall, or on a laptop
  • No ngrok, no reverse proxy, no cloud deployment needed for development
  • Slack pushes events over the WebSocket; your server handles them instantly
  • Uses the xapp- App-Level Token for the initial handshake

Slack Startup Sequence (slackApp.ts)

  1. Check that SLACK_SIGNING_SECRET, SLACK_APP_TOKEN, SLACK_BOT_TOKEN are set.
  2. Create a Bolt App instance with socketMode: true and the three credentials.
  3. Register Slack event handlers (via slackEventHandlers.ts).
  4. Call app.start() — opens WebSocket to Slack.
  5. Start DM Poller — 30-second interval timer begins.
  6. Start Channel Poller — 30-second interval timer begins.
  7. Call auth.test() to get the workspace ID and save it to the DB.

⚠️ Important: Channel Poller does not require bot invitation

Because the channel poller reads history using the user's own xoxp- token, it can see any channel the user is a member of — without ever inviting the bot to those channels. This is the key advantage over Slack event handlers which require the bot to be in the channel.

Context Building — How AI gets conversation history

Channel History (fetchHistory)
  • Calls conversations.history with latest = messageTs
  • Fetches last 15 messages before the new message
  • Resolves user IDs → real names for the AI
  • Filters out join/leave system messages
Thread History (fetchThreadHistory)
  • If message is a thread reply, calls conversations.replies
  • Fetches the full thread context
  • Excludes the triggering message itself
  • Sent separately as threadHistory to the AI prompt

AI Draft Generation

How OpenAI GPT-4o generates context-aware draft replies

07 — AI Draft Generation

Prompt Architecture

The AI is given a structured prompt that tells it who you are, who sent the message, what the conversation history is, and what the new message says. This multi-context approach produces far more relevant replies than single-message prompts.

Slack Prompt Structure (slackPromptService.ts) — simplified
// System prompt tells GPT who it's helping
const systemPrompt = `You are a helpful assistant that drafts Slack replies
for ${ownerName}. Write in a professional but friendly tone.
Keep replies concise (2-4 sentences). Do NOT sign off with a name.`;

// User prompt supplies the full context
const userPrompt = `
## Conversation History (${history.length} messages)
${history.map(m => `${m.senderName}: ${m.text}`).join('\n')}

## Thread Context
${threadHistory?.map(m => `${m.senderName}: ${m.text}`).join('\n') ?? 'N/A'}

## New Message From ${senderName}
"${messageText}"

Draft a reply from ${ownerName}'s perspective:
`;

// Single GPT-4o call with temperature 0.7
const response = await openai.chat.completions.create({
  model: 'gpt-4o',
  messages: [
    { role: 'system', content: systemPrompt },
    { role: 'user',   content: userPrompt },
  ],
  temperature: 0.7,
  max_tokens:  300,
});

⚠️ API Cost Awareness
  • Each draft = 1 GPT-4o API call. With filtering working correctly, only real human messages trigger calls.

  • The email filter and Slack dedup system ensure we never call GPT-4o for the same message twice

Database Schema

Prisma models, relationships, and deduplication strategy

08 — Database Schema

prisma/schema.prisma — Key Models
// Gmail user credentials
model Token {
  id           String   @id @default(cuid())
  userId       String   // Google email address
  accessToken  String
  refreshToken String
  expiresAt    DateTime
}

// Slack workspace — one per team
model SlackWorkspace {
  teamId       String   @id
  teamName     String
  botToken     String   // xoxb- — send DMs as bot
  botUserId    String
  userToken    String?  // xoxp- — read channels as user
  slackUserId  String?  // User's Slack ID (e.g. U0B3RB0GBFC)
  lastDMCheckAt DateTime? // Timestamp for DM poller
}

// Generated Slack drafts
model SlackDraft {
  id           String   @id @default(cuid())
  teamId       String
  channelId    String
  messageTs    String   // Slack message timestamp (unique ID)
  threadTs     String?
  fromUserId   String
  fromName     String
  originalText String
  draftText    String
  status       String   @default("pending")
  @@unique([messageTs, channelId]) // ← dedup key
}

// Email dedup — one record per processed Gmail messageId
model ProcessedEmail {
  id        String   @id @default(cuid())
  messageId String   @unique  // ← DB-level dedup
  userId    String
  createdAt DateTime @default(now())
}

// Email draft history
model DraftLog {
  id       String @id @default(cuid())
  userId   String
  threadId String
  subject  String
  draftText String
  createdAt DateTime @default(now())
  @@unique([userId, threadId]) // ← one draft per thread per user
}

Frequently Asked Questions

Does the bot ever send replies automatically without me seeing them?
No. The bot only creates drafts and sends them to YOU via a private DM. It never sends anything to the original sender on your behalf.
What happens if the same message is detected by both the event handler and the channel poller?
The deduplication system in slackMessageService.ts uses a unique (messageTs, channelId) constraint. The second attempt is silently skipped — only one draft is ever created per message.
Can multiple users connect their own Gmail and Slack accounts?
Yes. Each user authenticates via OAuth independently. Their tokens are stored per-user in the DB. The email poller loops over all connected users and processes each inbox separately.
Why does the channel poller use an in-memory Map instead of the database for timestamps?
Performance — checking the DB every 30 seconds for every channel would be expensive. The in-memory Map is fast and sufficient. On server restart, it falls back to lastDMCheckAt from the DB as a safe baseline.
How do I add a new email filter rule?
Open src/utils/emailUtils.ts and add a regex to either AUTOMATED_SENDER_PATTERNS (for sender addresses) or AUTOMATED_SUBJECT_PATTERNS (for subject lines). No other changes needed.
How do I extend this to support Microsoft Teams or Outlook?
Follow the same pattern: create a teamsPoller.ts for reading Teams messages (using Microsoft Graph API), a teamsDraftHelper.ts for AI generation and delivery, and add the poller to index.ts. The generateEmailReply() or generateSlackReply() AI function can be reused as-is.
